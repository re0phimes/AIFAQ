# TODO Prioritization and Task Governance Design

Date: 2026-03-20

## Context

当前项目同时维护两份任务文档：
- `Claude.md`：长期真源
- `todo.md`：讨论与执行视图

现有问题不是“没有 TODO”，而是：
- 当前重点和下一阶段筹备混在一起
- 排序缺少统一原则
- `todo.md` 与 `Claude.md` 的粒度不完全一致
- 后续要“一项一项推进”时，缺少一个可直接执行的全局顺序

用户已经确认本次需要一份“整体 TODO”，并且排序原则以“依赖关系和基础设施优先”为准。

## Confirmed Decisions

1. 输出一份单一总表，覆盖当前阶段与下一阶段。
2. 排序原则采用：
   - 先基础鉴权与执行隔离
   - 再审核控制面能力
   - 再质量增强与体验修复
   - 最后放平台扩展与下一阶段事项
3. 当前阶段新增一项：
   - 手机端查看自适应
4. 微信小程序支持不进入当前阶段，放入下一阶段。
5. 微信小程序相关数据库共享与免费额度风险评估必须作为该阶段的前置分析项保留。

## Scope

### In Scope

- 统一整理项目整体 TODO
- 给出明确执行顺序
- 区分当前阶段与下一阶段
- 为后续逐项推进提供稳定顺序
- 规范 `Claude.md` 与 `todo.md` 的同步方式

### Out of Scope

- 立即实现任一 TODO 项
- 立即接入微信小程序
- 立即做数据库迁移或容量扩容

## Recommended Ordering

1. `Admin API Key` 统一鉴权
2. Agent 触发与执行隔离
3. Admin Review 退回自动重生成
4. 后台批量 API（一期）
5. 历史信息查看
6. 开源脱敏（严格模式）
7. 相似问题识别（BERT，两段式）
8. 手机端查看自适应
9. 平台接入范围（一期）
10. 微信小程序支持（下一阶段）

## Ordering Rationale

### 1. Foundation First

`Admin API Key` 统一鉴权是所有程序化 admin 调用的共同前提。没有这层统一入口，后续 runner、批量操作、程序化上传都会出现权限模型分裂。

Agent 触发与执行隔离紧随其后，因为它决定了自动化能力是否能在安全边界内扩展。它不是单一功能点，而是后续自动重生成和外部接入的承载底座。

### 2. Control Plane Before Enhancements

退回自动重生成、批量 API、历史查看都属于 admin 审核控制面。它们之间的关系是：
- 自动重生成依赖鉴权和隔离执行链路
- 批量 API 依赖统一鉴权
- 历史查看增强审核闭环，但不阻塞前两个动作能力

因此控制面内部的顺序是：
- 先打通动作
- 再补齐批量化
- 最后补观察面

### 3. Safety Before Broader Expansion

开源脱敏排在平台继续外扩之前。原因很直接：当 runner、程序化调用、仓库开放度继续上升时，日志、任务 payload、敏感原文的暴露风险会被放大。先收紧脱敏策略，再继续扩展平台接入，风险更可控。

### 4. Quality and UX Follow Core Infra

相似问题识别和手机端自适应都重要，但它们不构成其他任务的基础依赖：
- 相似问题识别是质量增强能力
- 手机端自适应是用户体验修复项

它们都不应阻塞基础设施与控制面主链路，因此放在中后段。

### 5. Platform Expansion Last

一期平台接入范围本质是范围约束与集成策略，不是最早应该落地的底层能力。它应建立在前面的鉴权、隔离、脱敏和审核主链路稳定之后。

微信小程序支持明确属于下一阶段，因为它会引入：
- 新客户端形态
- 与现有后台共享数据库的边界问题
- 当前 Vercel Postgres 免费额度风险评估
- 可能的迁移或分层决策

这些不应该与当前阶段主线混排执行。

## Document Governance

建议采用以下写法：

- `Claude.md`
  - 保留“当前重点 TODO”作为真源
  - 新增“下一阶段 TODO”或“下一阶段筹备”
  - 用编号表达全局顺序

- `todo.md`
  - 以“整体 TODO（排序版）”作为执行视图入口
  - 明确每项状态：`todo` / `doing` / `blocked` / `done`
  - 若与 `Claude.md` 冲突，以 `Claude.md` 为准

## Success Criteria

- 项目存在一份用户认可的整体 TODO 顺序。
- 当前阶段与下一阶段不再混淆。
- 手机端自适应被纳入当前阶段。
- 微信小程序支持被明确降级到下一阶段。
- 后续可以按该顺序逐项展开设计与实现，不需要再次重排主线。
